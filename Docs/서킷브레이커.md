## 서킷 브레이커란?

MSA에서 각 서비스 간 호출 실패 상태를 감지하고 시스템의 전체 안정성을 유지하는 패턴입니다.

외부 서비스 호출 실패 시 빠른 실패를 통해 장애를 격리하고, 시스템의 다른 부분에 영향을 주지 않도록 합니다.

상태 변화는 Closed → Open → Half-Open 으로 이루어져 있습니다.

### Resilience4J란?

이는 서킷 브레이커 라이브러리로 서비스 간의 호출 실패를 감지하고 시스템의 안정성 유지를 도와줍니다.

다양한 서킷 브레이커 기능을 제공하며, 장애 격리 및 빠른 실패를 통해 복원력을 높여줍니다.

위에서 서킷 브레이커의 상태에 대해 이야기 했으니 각각이 무엇인지 자세히 살펴봅시다.

### Closed

기본 상태로 **모든 요청**을 통과시킵니다.

이 상태에서 호출 실패 시 실패 카운터가 증가합니다.

이 실패율이 설정된 임계값을 초과된다면 서킷 브레이커가 **Open** 상태로 전환됩니다.

예를 들어 최근 10번의 호출 중 7번이 실패하여 실패율이 70%에 도달하면 Open 상태로 전환됩니다.

### Open

서킷 브레이커가 오픈 상탤 전환되면 모든 요청을 즉시 실패로 처리합니다.

이 상태에서 요청이 실패하지 않고 바로 Error 을 반한합니다.

설정된 대기 시간이 지난 후 서킷 브레이커는 하프-오픈 상태로 전환됩니다.

### Half-Open

이 상태에서는 제한된 수의 요청을 허용해 시스템이 정상으로 복구되었는지 확인합니다.

요청 성공 시 서킷 브레이커는 Closed 로 상태가 전환됩니다.

만약 실패하면 서킷 브레이커는 다시 Open으로 상태가 전환됩니다.

### Failback

호출 실패 시 계속 error가 아닌 대체 로직을 제공해 시스템을 안정적으로 운영해야 합니다.

```java
@Service
public class MyService {
	@CircuitBreaker(name = "myService", fallbackMethod = "fallbackMetod")
	public String myMethod() {
		// 외부 서비스를 호출한다.
		return externalService.call();
	}
	
	public String fallbackMethod(Throwable t) {
		return "fallback response";
	}
}
```

이를 통해 시스템을 안정적으로 운영하고 장애가 발생해도 사용자에게 일정한 응답을 제공할 수 있다.

또한 장애가 전파되는 것을 막을 수 있다.

### 모니터링

서킷 브레이커 상태를 모니터링하고 관리할 수 있는 도구를 제공합니다.

Resilience4j Dashboard 를 통해 가능합니다.

```groovy
implementation 'io.github.resilience4j:resilience4j-micrometer'
implementation 'io.micrometer:micrometer-registry-prometheus'
implementation 'org.springframework.boot:spring-boot-starter-actuator'
```

위에서부터 순차적으로 설명하면

1. Resilience4j의 서킷 브레이커, 재시도, 속도 제한 등 기능을 Micrometer가 이해할 수 있는 데이터로 변환해 줍니다. (Metrics 수집 자동화 툴)
2. 이 metric 데이터를 prometheus 로 변환해 줍니다. (Prometheus 는 모니터링 서버로 초 단위로 수집해 DB에 저장해준다. 그래프 등을 볼 수 있다.)
3. actuator은 상태를 엔드포인트로 제공해 관리가 편하다.

다음의 환경 설정을 해주면 적용 완료!

```yaml
management:
	endpoints:
		web:
			exposure:
				include: prometheus
	prometheus:
		metrics:
			export:
				enabled: true
```

이를 이용하면 http://${hostname}:${port}/actuator/prometheus 에 접속해 서킷 브레이커 항목을 확인 가능하다.

하지만 이만 사용하면 Raw data 만 보이게 되는데, 이를 위한 툴이 바로 그라파나 이다.

(프로메테우스와 그라파나는 거의 같이 사용하게 됩니다.)

`/acturaor/prometheus` 로 접속하면 다음 같이 보이게 되는데,

```yaml
# HELP resilience4j_circuitbreaker_states Circuit Breaker States
# TYPE resilience4j_circuitbreaker_states gauge
resilience4j_circuitbreaker_states{name="backendA",state="closed",} 1.0
resilience4j_circuitbreaker_states{name="backendA",state="open",} 0.0
resilience4j_circuitbreaker_states{name="backendA",state="half_open",} 0.0
```

이를 통해 사람이 보기는 아주 어려울 것 입니다.

이를 Grafana를 사용하면 대시보드를 통해서 보여줍니다.

저는 모두 docker 환경에서 사용하기에

`docker run -d -p 3000:3000 grafana/grafana` 를 통해 열어두겠습니다.

이후 http://localhost:3000 에 접속합니다. (IP와 PW는 admin/admin 입니다.)

그 이후 Connections → Data Source → Add data source → Prometheus 를 택한 후 연결하면 됩니다!

### 실습

<aside>
💡

spring starter가 아닌 github.resilience4j:resilience4j-spring-boot3:2.2.0을 사용할 것 입니다.

</aside>

[](https://mvnrepository.com/artifact/io.github.resilience4j/resilience4j-spring-boot3)

~~-라고 썼지만 저는 스프링 부트 4 를 쓰기에 spring-boot4를 사용했습니다.-~~

이유는 Spring Cloud 에서 제공하는 프레임워크는 추상화 계층만 제공하고 Resilience4J 에서 제공하는 스타터의 경우 Retry, RateLimiter, Bulkhead 등 기능을 지원합니다.

AOP도 같이 추가합시다.

설정 파일은 다음과 같이 합니다.

```yaml
resilience4j:
	circuitbreaker:
		configs:
			default:
				registerHealthIndicator: true # app의 health check 에 서킷 브레이커 추가
				slidingWindowType: COUNT_BASED # 슬라이딩 윈도우 타입 설정
					# 만약 COUNT_BASED 라면 마지막 N 번 호출 결과를 기반으로 결정하고
					# TIME_BASED 라면 마지막 N초 호출 결과를 기반으로 결정합니다.
				minumNumberOfCalls: 5 # 서킷 브레이커 동작 최소 호출 수
				slowCallRateThreshold: 100 # 느린 호출 비율이 100%를 도달하면 서킷 브레이커 동작
				slowCallDurationThreshold: 60000 # 느린 호출 기준 시간(밀리초), 60초 이상이면 동작
				failureRateThreshold: 50 # 실패율이 이 임계값을 ㅊ과하면 동작
				permittedNumberOfCallsInHalfOpenState: 3 # Half-Open 상태에서 허용하는 최대 호출 수
				waitDurationInOpenState: 20s # open -> half-open 전환되기 전 대기하는 시간
```

### 실습 적용

```java
@GetMapping("/{productId}")
    public ResponseEntity<ProductResponse> getProduct(@PathVariable("productId") Long productId) {
        log.info("[getProduct]Current Product-App Port is" + port);
        return ResponseEntity.ok(productService.getProduct(productId));
    }
```

현재 있는 이 코드를 수정해서 만약 현재 없는 product Id가 들어오면 예외처리 후 failback 처리 과정까지 보려 합니다.

```java
@Override
    public ProductResponse getProduct(Long id) {
        Product product = findProductOrThrow(id);
        return ProductResponse.from(product);
    }
```
