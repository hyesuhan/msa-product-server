## MSA 란?
---

### Micro-Service Architecture (MSA)

하나의 애플리케이션을 여러 독립적인 서비스로 분리해 개발, 배포, 유지보수를 용이하게 하는 **SW 아키텍처 스타일**입니다.
각 서비스는 특정 비즈니스 기능을 수행하며, 서로 독립적 배포 및 확장 가능합니다.
서비스 간 통신은 주로 HTTP/HTTPS, 메시지 큐 등을 통해 이루어집니다.

### 특징

독립적인 배포 가능
    각 서비스는 독립적으로 배포할 수 있으며, 다른 서비스에 영향을 주지 않고 업데이트 가능합니다.

작은 팀으로 구성
    각 서비스는 작은 팀이 독립적으로 개발하고 관리 가능합니다.

기술 스택의 다양성
    각 서비스는 적절한 기술 스택을 자유롭게 선택 가능합니다.

## 모놀리틱 아키텍처와 비교

### Monolitic Architecture 장단점

모놀리틱 아키텍처는 하나의 큰 코드베이스로 구성된 애플리케이션입니다.
모든 기능이 하나의 애플리케이션 내에 포함됩니다.

**[장점]**

간단한 배포와 단일 DB로 개발과 운영에 용이해 초기에 빠른 개발이 가능합니다.

**[단점]**

긴 개발 주기, 확장성 부족(특정 기능 확장 시 전체 애플리케이션을 확장해야 함), 유연성 부족


### Micro-Service Architecture 장단점

여러 개의 독립 서비스로 구성된 애플리케이션
각 서비스는 특정 비즈니스 기능을 수행한다.

**[장점]**

확장성(특정 서비스만 확장하여 성능 최적화가 가능하다.)

독립적 배포(개별 서비스의 변경 사항을 독립적으로 배포 가능)

유연성(서비스별 적합 기술 스택 선택 가능)

**[단점]**

복잡성 증가(서비스 간 통신, 데이터 일관성 유지 등 복잡성)

운영 비용 증가(각 서비스 모니터리, 로깅 등을 개별적으로 관리 해야 함)

데이터 관리(분산 DB 로 인해 데이터 일관성 유지가 어려움)

네트워크 지연(서비스 간 통신이 네트워크를 통해 이뤄지기에 지연 시간 발생 가능)

그렇기에 항상 MSA를 쓰는 것이 유용한 것이 아닌, 서비스 규모 및 팀 구성 등에 따라 적절히 선택해야 합니다.
