### 서버 사이드 로드 밸런싱이란?
- 클라이언트와 서버 사이 별도 장비가 필요하다. Nginx가 예시다.
- 로드 밸런서가 대상 서버 리스트르 결정하며 클라이언트가 VIP(Virtual IP)로 요청 후 Load Balancer가 분산 후 전달합니다.
- 장점으로는 Client가 서버 구조를 몰라도 되지만, LB 자체가 병목 지점이 될 수도 있습니다.
- AWS ELB, Nginx가 이에 해당합니다.

### 네트워크 계층에 따른 로드 밸런서
#### L4(Transport Layer) 로드 밸런싱
- **IP 주소와 포트 번호(TCP/UDP)를 기반**으로 트래픽을 분산합니다.
- 패킷 내용을 열어보지 않기에 **속도가 빠르고 효율적**입니다

#### L7(Application Layer) 로드 밸런싱
- HTTP , Header, Cookie, URL 경로, Query String 등 **실제 요청 내용을 기반으로 분산**합니다.
- 특정 URL 패턴에 따라서 다른 서버로 **라우팅하는 등 정교한 제어**가 가능합니다.
- 예를 들어 MSA에서 API GateWay가 될 수 있겠네요.

  API GateWay는 L7 로드 밸런싱 기술을 활용해 라우팅을 수행하는 구조입니다!

  그러면 API gateway도 로드 밸런서냐고 물을 수도 있는데, 이게 좀 역할이 비슷해진 것 뿐입니다.

    - 로드 밸런서는 “***들어온 요청을 공평하게 나눠주는 것***” 이고
    - API Gateway는 “***시스템의 문자기***로 인증, 필터링, 라우팅을 하는데, 라우팅을 하려다 보니 ***뒤에 서버가 여러 대여서 자연스레 로드 밸런싱 기능을 가져야 겠다.(Has-A)***”
    - 의 느낌인 것입니다!!

### 그런데 굳이 코드 2갤 관리해야 하는 이유가 있나요?
> 그냥 Replica(복제본)으로 띄우면 되는거 아니야?

인프라(K8s,ECS 등)에서 Replica를 띄우는 것은 쉬운데 애플리케이션이 늘어난 이 Replica이 주소를 알기가 쉽지 않습니다.

새로운 Replica가 생길 때마다 Nginx 같은 로드 밸런서의 설정 파일에서 주소를 일일이 적어줘야 합니다.

또한 서버가 죽으면 수동으로 빼야 합니다.

만약 위처럼 코드로 관리하게 된다면, Replica의 서버의 IP가 변할 때(Auto Scaling) 자동으로 등록됩니다.

또한 호출하는 쪽은 코드를 수정할 필요 없이 @FeignClient 를 통해서 실행 시점에 주소를 알아서 가져오게 됩니다.

### 그렇다면 이렇게 코드로 관리하는 것의 단점도 있어 보이는데..

코드로 관리하면 유연하지만 대신 다음의 문제가 있다.

언어의 종속성, 클라이언트의 비대화(코드가 무거워짐), 설정 동기화 어려움이 있을 수 있음

### 코드로 관리하면 Product 1 이 수정되면 2 도 수정되어야 하는데 어떻게 관리하나요?
> 사[..](..%2F..)람 손으로 하는건 당연히 실수도 있고 문제도 있을 수 있을 것 같은데…

1. 이미지 기반 관리(Docker)

   코드 수정 시 서버에 접속해서 고치는 것이 아니라 그냥 새로운 Image로 만들면 됩니다.

2. CICD 파이프라인을 통해 순차적으로

   2개의 인스턴스를 쓰는 것은 로드 밸런싱도 있을 수 있지만, 버전 교체일 수도 있습니다.

   만약 2개 다 동일 업데이트를 했는데 HotFix가 발생할 수도 있죠?

   그렇기에 Doker 이미지를 생성해 Product1, 2를 순차적을 교체할 수 있습니다. (Rolling Update)

   이 고정에서 Eureke가 구 버전 주소를 지우고 신버전으로 자동 등록됩니다.
