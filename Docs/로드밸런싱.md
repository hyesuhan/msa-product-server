## 03. 로드 밸런싱

### 로드 밸런싱이 무엇인가요?

로드 밸런싱은 네트워크 트래픽을 여러 서버로 분산시켜 서버의 부하를 줄이고, 시스템의 성능과 가용성을 높이는 기술입니다.

서버 간 트래픽을 고르게 분배해 특정 서버에 부하가 집중되는 것을 방지해줍니다.

로드 밸런싱을 하는 기술은 다양하게 있는데 크게 **누가 결정을 하는가**와 **어느 계층에서 동작하는가**가 있습니다.

이에 대한 종류로 Server-Side, Client-Side, Transport, Application Load Balancer가 있습니다.

하지만 저희는 이를 Application 내부 라이브러리에서 제어할 것이기에 **Client Side**에 대해 조금만 알아보겠습니다.

### 클라이언트 사이드 로드 밸런싱이란 무엇인가요?

따로 외부에 로드 밸런서가 있는 형태가 아닌, Client가 직접 여러 서버 중 하나를 선택해 요청을 보내는 방식입니다.

클라이언트는 서버의 목록을 가지고 있어서 이를 바탕으로 로드 밸런싱을 수행합니다.

즉, **부하 분산 알고리즘**이 실행되는 위치가 바로 **Application 내부**이고 어디로 보낼지를 결정하는 것은 바로 자기 자신입니다!

### FeignClient 란?

Spring Cloud 에서 제공하는 **HTTP 클라이언트**로, **선언적**으로 **RESTful 웹 서비스를 호출**할 수 있습니다. (RestTemplate과 비슷하지만 **선언적**이 중요. 기억이 안 난다면 이전 chapter 다시 보고 오기!)

Eureka와 같은 서비스 디스커버리와 연동해 동적으로 **서비스 인스턴스를 조회**하고 **로드 밸런싱을 수행**할 수 있습니다.

- 인터페이스와 Annotation으로 **REST API를 호출**할 수 있습니다.(←이게 바로 **선언적**의 의미 입니다)
- Eureka와 통합해 서비스 인스턴스 목록을 동적으로 조회하고 로드밸런싱을 수행합니다.
- Ribbon이 통합되어 있기에 **자동으로 로드 밸런싱**을 수행시켜 줍니다.

### Ribbon 이란?

Netflix 개발 Client-Side 로드 밸런서로, MSA 인스턴스 부하 분산 프레임워크입니다.

다양한 로드 밸런싱 알고리즘을 지원(RR, 가중치 기반 등)하고 Eureka 같은 서비스 디스커버리와 연동해 사용할 수 있습니다.

또한 요청 실패 시 다른 인스턴스로 자동 전환하는 FailOver도 지원합니다.

`FeignClient` 가 `@FeignClient`  어노테이션을 통해 가상 클라이언트를 만들면 (Spring JPA 처럼 HTTP 요청을 추상화합니다.)

Ribbon에서 위 요청에 따라 Eureka에서 주문 서비스의 IP 리스트를 가져와서 그 중 하나를 택하게 합니다.

**❓동작 메커니즘이 어떻게 될까요?**

1. Feign가 인터페이스에 정의된 경로와 파라미터를 읽어서 HTTP 요청 본문을 만들게 됩니다.
2. 요청에 적힌 서비스 이름(order-service)를 Ribbon이 식별하게 됩니다.
3. Ribbon이 부하 분산 알고리즘을 통해 실제 서버 주소로 치환하여 Load Balancing을 해줍니다.
4. 이를 통해 결정된 주소로 실제 패킷을 보냅니다.

### FeignClient와 Ribbon 설정 방법

일단은 의존성을 당연히 추가해야 겠죠?
```groovy
implementation 'org.famework.cloud:spring-cloud-starter-netflix-eureka-client'
implementation 'org.framework.cloud:spring-cloud-starter-openfeign'
```

spring boot 애플리케이션에 다음 @EnableFeignClient 설정을 통해 웹 서비스 호출을 가능하게 됩니다.
```groovy
@SpringBootApplication
@EnableFeignClients
public class MyServiceApplication {
    public static void main(String[] args) {
        SpringApplication.run(MyServiceApplication.class, args);
    }
}
```

위에서 FeignClient 는 선언적으로 호출할 수 있다고 했으니 다음과 같이 서비스 호출을 할 수 있습니다.
```groovy
@FeignClient(name = "order-service")
public interface OrderServiceClient {
	@GetMappin("/endpoint")
	String getResponse(@RequestParam(name="param") String param);
}
```
### 로드 밸런싱 알고리즘

위에서 로드 밸런싱 알고리즘을 선택해 사용할 수 있다고 언급했습니다.

**↪️ 라운드 로빈 (RR)**

각 서버에 **순차적으로 요청을 분배**합니다.

만약 product-1, product-2, product-3 이 있다면 product에 요청이 들어올 때마다  1→2→3 순으로 요청을 보내게 됩니다.

간단하고 공평하게 트래픽을 분산하게 됩니다.

**➕ 가중치 기반 로드 밸런싱**

각 서버에 **가중치를 부여**해서 이에 비례하여 요청을 분배하게 됩니다.

만약 각 인스턴스의 성능이 다르거나 네트워크 상태가 다르다면 가중치를 조절해 분산할 수 있습니다.

**💁‍♂️ 그 외의 알고리즘**

**최소 연결**: 현재 연결된 클라이언트 수가 가장 적은 서버로 요청을 보냅니다.

**응답 시간 기반**: 서버의 응답 시간을 기준으로 가장 빠른 서버로 요청을 보냅니다.

### 예시로 살펴보기

현재 다음의 서버가 있습니다.

Eureka Server

User Service

Order Service

Product1, Product2 Service

**1️⃣ 서비스를 등록하겠습니다.**

`Product1` 와 `Product2` 가 실행될 때 자신의 IP와 포트를 Eureka에 등록하지만 서비스 이름은 둘 다 동일하게 `product-service` 입니다.

**2️⃣목록을 갱신합니다.**

`Order Service`가 시작되며 Eureka에서 `product-service`인스턴스를 리스트로 가져와 로컬 캐시에 저장하게 됩니다.

**3️⃣ FeignClient 정의를 합니다.**

Order Service 내부에는 다음의 인터페이스가 존재합니다.

```groovy
@FeignClient(name = "product-service")
public interface ProductClient{

	@GetMapping("/products/{id}")
	ProductResponse getProduct(@PathVariable("id") Long id);
}
```

**4️⃣ Ribbon을 통해 로드 밸런싱이 수행됩니다.**

`User`가 `Order Service`에 주문 요청을 보냅니다.

`Order Service`가 `productClient.gtProduct(1L`) 을 호출하면, 내부의 Ribbon이 동작됩니다.

1. 목록을 확인합니다. 👁️: `product-service`에 해당하는 IP 리스트를 확인합니다.
2. 알고리즘이 적용됩니다.: RR에 따라서 첫 번째 요청은 `Product1`, 두 번째 요청은 `Product2` 로 갑니다.
3. 치환: 가상의 서비스명인 `product-servie`를 실제 물리 주소(http://localhost:8081)로 바꿉니다.

**5️⃣ 실행 및 응답이 됩니다.**

결정된 주소로 HTTP 요청을 보내고 `Product1`로 부터 받은 데이터를 `Order Servie`가 가공해 `User`에서 최종 응답을 전달합니다.
