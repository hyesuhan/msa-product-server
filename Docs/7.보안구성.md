### 보안의 중요성

MSA에서는 특히 각 서비스가 독립적으로 배포되고 통신하기에 보안이 중요합니다.  


### OAuth2란?  


이는 Protocol(통신) 입니다.

제 3 서비스(Google, Kakao 등)에 권한을 요청하고 승인받는 절차를 정의한 표준 프로토콜입니다.  


4가지 역할을 가지고 있습니다.

1. Resource Owner: 사용자
2. Client: 우리가 만드는 Application
3. Authorization Server: 권한을 부여해 주는 서버(Google, Kakao, etc)
4. Resource Server: 실제 데이터가 있는 서버 (Google 마이페이지, Kakao Profile API)

사실 적용할 때는 각 서비스에서 제공하는 문서를 읽고 도입하는 것이 가장 효율적입니다.  


[OAuth 2.0을 사용하여 Google API에 액세스하기  |  Authorization  |  Google for Developers](https://developers.google.com/identity/protocols/oauth2?hl=ko)  


[Kakao Developers 문서 v2](https://developers.kakao.com/docs/ko/kakaologin/rest-api)  


또한 개인적으로 이 Oauth2 를 많이 개발 및 참고 자료를 보면서 개인마다 디자인 패턴이나 적용 방식이 매우 다름을 좀 알았습니다.  
저의 경우는 여러 OAuth의 경우 전략 패턴을 사용하고 추가적으로 일반 로그인도 적용하는 경우 통일화된 인증 객체로 상속하는 것을 선호합니다.  
또한 저는 일관된 패턴을 좋아하기에 토큰 발급도 저희 쪽에서 하는 것을 더 선호합니다.(Implicit Grant)  

  

### JWT란?

JSON Web Token 으로 인증에 필요한 정보를 JSON 객체에 담아서 암호화된 토큰의 한 종류입니다.

OAuth2.0 에서 발급되는 `AccessToken` 을 만들 때 가장 많이 사용되는 형식입니다.  


### JWT 구조

이는 점(`.` )으로 구분된 세 부분이 있습니다.

1. Header: 토큰의 타입과 암호화 알고리즘이 무엇인지
2. Payload: 실제 담고 싶은 정보(사용자 ID, 만료 시간 등, 이를 Claim 이라 합니다)
3. Signature: 서버만 알고 있는 비밀키로 생성한 서명으로 위변조를 방지합니다.  



따라서 우리는 이 토큰이 있다면 Payload 를 통해 사용자 ID 같은 정보를 바로 추출할 수 있습니다.  


**단, 아주 중요한 것은 노출되면 안되는 정보는 담으면 안됩니다.**  


우리 뿐 아니라 남도 볼 수 있기 때문입니다.  


또한 JWT는 서버가 저장하지 않는 Stateless 이기 때문에 한 번 발급 시 만료 전까지 강제 로그아웃이 어렵습니다.

그래서 일반적으로 만료 기간이 짧은 `AccessToken` 과 `RefreshToken` 을 함께 사용해 AccessToken이 만료되면 RefreshToken으로 재발급해줍니다.  


Redis는 이 토큰 관리를 위한 외부 저장소입니다. 이는 만료 기간에 따라 TTL을 조절할 수 있어 유용합니다. 서버 입장에서는 Redis 처럼 조회만 할 뿐, 서버 자체가 사용자 상태를 알고있지는 않습니다.

이를 사용하면 중복 로그인 방지, 서버 강제 로그아웃 등의 제어를 할 수 있습니다.

따라서 이 Redis가 죽었을 때를 대비한 FailOver이나 적절한 만료 기간 등에 대해 전략을 세울 수 있습니다.  



### Dependency

다음을 추가합니다. 이는 JWT Token을 위한 의존성입니다. OAuth2 의 경우 이와 관련된 OAuth2 를 넣어야 합니다.

```groovy
implementation 'io.jsonwebtoken:jjwt:0.12.6'
implementation 'org.springframework.boot:spring-boot-starter-security'
```
